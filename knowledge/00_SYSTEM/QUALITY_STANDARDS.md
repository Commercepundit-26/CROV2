# Quality Standards

## Purpose
Defines the writing style, depth, and formatting expectations for the repository.

## Writing Style
- Direct, factual, and concise. No fluff.
- Use bullet points over long paragraphs.
- Active voice (e.g., "The engine generates the PPT" not "The PPT is generated by the engine").

## Technical Depth
- Must contain actual variable names, API endpoints, and exact file paths where applicable.

## Business Depth
- Must tie technical implementations back to CRO objectives (e.g., "We take screenshots because visual proof increases client trust in the audit").

## Examples & Formatting
- Code examples must be enclosed in triple backticks with the correct language tag.
- JSON must strictly match the schemas in `09_SCHEMAS`.

## Acceptance Criteria
A document is only considered "High Quality" when a new engineer can implement its requirements without asking clarifying questions.
